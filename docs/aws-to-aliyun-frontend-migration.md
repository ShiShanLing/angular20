## 1. 迁移前：摸清改动范围

在代码与配置中全局搜索，列出**所有需要替换的地址**（建议用表格记录「旧值 → 新值 → 文件路径」）。

| 类型 | 常见关键词 / 写法 | 阿里云侧常见对应 |
|------|-------------------|------------------|
| API 基址 | `API_URL`、`baseURL`、`VITE_*`、`REACT_APP_*`、`EXPO_PUBLIC_*` | 新域名 / SLB / API 网关 |
| 静态资源 | `cdn.xxx.com`、S3 完整 URL | OSS + CDN 域名 |
| WebSocket | `wss://` | 新网关或 SLB |
| 文件上传 | S3 presigned URL | OSS 签名 URL（字段可能不同） |
| OAuth / 支付 | `redirect_uri`、`return_url` | 各平台重新登记域名 |
| 地图 / 短信 / 推送 | SDK key、域名白名单 | 按厂商文档更新 |

**原则**

- 环境相关配置走**环境变量**，不要散落在业务组件里硬编码。
- 先对 **staging / 测试域名** 打一包验证，再上生产。

### 搜索命令参考

```bash
# 在仓库根目录执行，按需调整排除目录
rg -i "amazonaws|cloudfront|s3\.|aws_" --glob '!node_modules' --glob '!dist' --glob '!.git'
rg -i "API_URL|BASE_URL|baseURL|VITE_|REACT_APP_|EXPO_PUBLIC_" --glob '!node_modules'
rg "https?://" --glob '*.{ts,tsx,js,jsx,vue,env*}' --glob '!node_modules'
```

## 2. 域名与 DNS

- [ ] 确认 **API 域名**、**静态资源域名**、**管理后台域名** 是否拆分（建议拆分，便于 CDN 与缓存策略）。
- [ ] **DNS 解析**切到阿里云前，将 TTL 提前调短（如 300s），减少「一半用户仍指向 AWS」的窗口期。
- [ ] **HTTPS 证书**在新域名 / 新 CDN 上部署完成，并验证证书链完整。
- [ ] **Cookie / Session**：`Domain`、SameSite、是否跨子域；域名变更后登录态是否失效，是否需要兼容期。
- [ ] **CORS**：后端在新域名上配置 `Access-Control-Allow-Origin`；若前端 `credentials: 'include'`，Origin 不能为 `*`。
- [ ] **HSTS / 301**：旧域名是否长期 301 到新域名（书签、SEO、App 内 WebView）。
- [ ] **ICP 备案**（国内自有域名）：备案、CDN 加速域名是否满足监管要求（与纯 AWS 海外部署差异大）。

## 3. 静态资源与 CDN

- [ ] **S3 → OSS**：Bucket 名、路径规则、公有读 vs 私有、签名 URL 逻辑（通常需后端配合改接口）。
- [ ] **CloudFront → 阿里云 CDN**：缓存规则、`index.html` 建议**不缓存**或极短缓存。
- [ ] 发布后执行 **CDN 刷新/预热**，避免用户仍加载旧 JS。
- [ ] 构建配置 **publicPath / assetPrefix**（Webpack `publicPath`、Vite `base`）与新 CDN 域名一致，避免 chunk、字体、图片 404。
- [ ] SPA **history 模式**：404 回退到 `index.html` 已在 OSS/CDN/网关配置。

---

## 4. 构建与 CI/CD

- [ ] CI 环境变量、密钥、部署脚本从 AWS CLI / `s3 sync` 改为 **ossutil / 阿里云 CLI** 或新流水线。
- [ ] **dev / test / prod** 三套变量分离，确认本次构建使用的是哪套 `.env`。
- [ ] **Source map** 上传路径变更后，Sentry 等错误平台的 source map 地址同步更新。
- [ ] 构建产物使用 **content hash 文件名**；若 CDN 强缓存了 `index.html`，会出现「接口已新、前端仍旧」。

## 5. 与后端对齐的契约

迁移前向后端 / 运维确认（前端不要假设「只换域名」）：

- [ ] API **路径**是否不变（仅换域名 vs 网关路径也变更）。
- [ ] **鉴权**：JWT 签发方、Cookie 名、refresh 接口是否变更。
- [ ] **文件上传**：直传 OSS 的 policy / signature 字段是否与 S3 不同。
- [ ] **WebSocket / SSE**：端点 URL、是否要求统一走 443 / `wss`。
- [ ] **WAF / 限流**：新域名是否被安全策略误拦。

## 6. 第三方与域名白名单

以下配置常在**第三方控制台**，不在业务仓库里，但必须在新域名下重新登记：

- [ ] 微信 / 支付宝 / 银联：授权回调域名、JSAPI 安全域名
- [ ] OAuth（Google、GitHub、企业微信等）：Redirect URI
- [ ] 短信、邮件模板中的链接域名
- [ ] 地图、人机验证、统计（GA、神策等）站点 ID / 域名
- [ ] **CSP**（Content-Security-Policy）：`connect-src`、`img-src`、`script-src` 是否仍包含旧 AWS 域名

---

## 7. 移动端 / 混合 App

若包含 App、Expo、React Native、Cordova 等：

- [ ] **Universal Links / App Links**：`apple-app-site-association`、`assetlinks.json` 托管域名变更
- [ ] **Deep Link / URL Scheme**：关联的 `https://` 域名是否更新
- [ ] **推送** payload 中的 H5 链接域名
- [ ] **热更新 / OTA** 包下载地址（若原指向 S3）
- [ ] **应用商店**：隐私政策、审核说明中的 API 域名是否需随版本更新

---

## 8. 国内环境与网络

- [ ] 国内用户访问：静态资源走 **OSS + 国内 CDN** 通常优于 CloudFront
- [ ] 若部分服务仍留在 AWS：评估国内访问稳定性，是否需要双栈（国内阿里云 + 海外 AWS）
- [ ] CI 构建机若在境内：npm / Docker 镜像源策略是否需调整

## 10. 本地操作步骤（摘要）

1. 向运维/后端索取：**新域名表、环境变量表、OSS/CDN 说明、CORS 是否已配置**。
2. 修改 `.env.*`、CI 变量、`publicPath` 等，避免在组件内散落 URL。
3. 使用**目标环境**变量执行构建（`npm run build` / `eas build` 等）。
4. 部署到 OSS，刷新 CDN，用新域名做全链路测试。
5. 按发布计划切换 DNS 或灰度流量，观察监控与 4xx/5xx。

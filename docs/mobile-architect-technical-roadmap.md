# 移动端架构师技术路线（详细版）

本文档依据 [`mobile-architect-requirements.md`](./mobile-architect-requirements.md) 中的能力模型，整理为 **可执行的学习与成长路线**。  
时间仅为 **粗粒度参考**（全职学习与工作中的累积速度差异极大）；可按「主修一端 + 选修另一端/跨端」裁剪。

---

## 如何使用本文档

1. **选定主轴**：Android **或** iOS 其一作为「精通端」，另一条轴用「能评审、能对接」为目标即可（除非你要双原生专家）。  
2. **按阶段推进**：每阶段有 **学习目标 → 实践清单 → 产出物 → 自测标准**；建议至少完成 **80% 实践清单** 再进入下一阶段。  
3. **与工作对齐**：优先把学习绑定到 **当前业务**（性能、模块化、监控），成长最快。  
4. **文档联动**：岗位要求中的「架构职责」「面试考点」在下列阶段中均有对应里程碑。

---

## 路线总览（阶段与侧重）

| 阶段 | 大致年限参考（移动端专职） | 核心主题 |
|------|---------------------------|----------|
| **L1 入门** | 0–1 年 | 语言与 SDK、UI 与生命周期、调试与发布入门 |
| **L2 进阶** | 1–3 年 | 架构范式、网络与存储、异步与并发、测试基础 |
| **L3 资深** | 3–6 年 | 模块化、性能与稳定性、工程化与 CI/CD、跨端/另一端广度 |
| **L4 架构** | 6–10 年+（因人/平台而异） | 选型与演进、RFC/ADR、可观测性与复杂域、协作与带人 |

以下为 **逐阶段详解**（含 Android / iOS 分叉要点）。

---

## L1 入门（0–1 年）：平台与交付闭环

### L1 学习目标

- 熟练使用 **官方 UI 框架** 完成完整页面与导航。
- 理解 **应用生命周期、进程与任务栈**（Android Activity/Fragment 或 iOS ViewController/SwiftUI scene）。
- 能独立 **打包、签名、安装到真机**，理解 Debug / Release 差异。
- 会用 **断点、日志、基础性能视图** 定位简单问题。

### L1 — Android 路径

| 主题 | 学习内容 | 实践清单 |
|------|----------|----------|
| 语言 | Kotlin 语法、空安全、集合与扩展 | 用 Kotlin 重写一个小 Demo 的全部 Java 片段（若有） |
| UI | Activity、Fragment、ViewBinding / Compose 入门 | 实现「列表 + 详情」导航；适配深色模式 |
| 生命周期 | onSaveInstanceState、配置变更、后台限制 | 旋转屏幕后状态不丢；理解「不要在 Activity 存重状态」 |
| 异步 | Coroutines 基础、Main 线程约束 | 网络请求不在主线程；cancel 与生命周期绑定 |
| 构建 | Gradle module、buildTypes、ProGuard/R8 概念 | 打出 debug / release 包；读一遍 `build.gradle.kts` |
| 发布 | 签名、ApplicationId、权限清单 | 写清楚 `AndroidManifest` 权限用途 |

### L1 — iOS 路径

| 主题 | 学习内容 | 实践清单 |
|------|----------|----------|
| 语言 | Swift 基础、Optional、协议与泛型入门 | 小型 CLI 或 Playground 熟练语法后再进 UIKit/SwiftUI |
| UI | UIKit Auto Layout **或** SwiftUI 声明式布局 | List + Navigation；Dark Mode |
| 生命周期 | UIViewController、scene、SwiftUI `@State` | 理解何时重建视图；内存里勿囤积大对象 |
| 并发 | `async/await`、MainActor | 网络回调不阻塞 UI |
| 工程 | Xcode Scheme、Target、签名与 Capabilities | 真机调试；理解 Debug/Release |
| 上架要点 | Info.plist、隐私描述文案占位 | 列出相册/定位等权限用途（即便 Demo） |

### L1 产出物（ portfolio 建议）

- 一个 **含网络请求的 CRUD 小应用**（列表展示远程数据 + 本地缓存可选）。
- README：如何构建、已知限制、截图。

### L1 自测标准

- [ ] 能向同事 **口述** 一次「从点击图标到首屏显示」的关键路径。  
- [ ] 能解释 **为什么主线程不能做耗时 IO**。  
- [ ] Release 包可在 **非本人电脑** 按 README 构建（或 CI 可构建）。

## L2 进阶（1–3 年）：架构直觉与工程质量

### L2 学习目标（对照岗位要求 §2.3 / §2.5 基础）

- 在项目中落地 **MVVM（或等价单向数据流）**，区分 View / 状态 / 副作用。
- 搭建 **可测试的网络层**：拦截器、错误模型、重试与超时策略（适度）。
- 掌握 **依赖注入**（Hilt / Swinject / 手写工厂）中的一种。
- **单测**：ViewModel / UseCase 层关键逻辑有测试；理解 fake vs mock。

### L2 — 架构与设计

| 主题 | 要点 | 实践 |
|------|------|------|
| 分层 | Presentation / Domain / Data（或等价） | 把「解析 JSON」移出 ViewModel；定义 Repository 接口 |
| 状态 | UI State 单一数据源、不可变更新 | 大屏旋转或进程死亡恢复策略 |
| 路由 | 深层链接需求预判 | 统一路由表或 Navigation 图（一端即可） |
| DI | 构造注入、作用域（Activity/Singleton） | 替换全局单例为新模块注入 |

### L2 — 网络与安全基础（对照 §2.3 / §2.6）

| 主题 | 要点 | 实践 |
|------|------|------|
| TLS | 证书链、MITM 风险认知 | Charles 调试时理解信任证书含义 |
| 存储 | Keychain / EncryptedSharedPreferences | Token 不落明文 SharedPreferences |
| API 契约 | 版本化、向后兼容 | 与后端约定「新增字段默认兼容」 |

### L2 — 测试（对照 §2.5）

- **单元测试**：Repository 用 fake datasource；ViewModel 测 reducer 逻辑。  
- **仪器/UI 测试**：一条冒烟路径（登录或核心 Tab）。  
- 明确：**不测布局像素**，测 **行为与状态**。

### L2 — Android 专项加深

- Jetpack：ViewModel、LiveData/StateFlow、Room（或 DataStore）、Navigation、WorkManager 选用场景。  
- **内存**：LeakCanary 引入；理解 Bitmap 与列表回收。

### L2 — iOS 专项加深

- Combine 或 AsyncSequence 与 UI 绑定（择一深入）。  
- Core Data / SwiftData / GRDB 等 **选型对比**（会做 POC）。  
- Instruments：**Leaks / Allocations** 入门。

### L2 产出物

- 给现有模块写 **1 份架构说明**（1～2 页）：分层图 + 数据流。  
- **10+ 个单测** 覆盖核心业务分支。

### L2 自测标准

- [ ] 能画 **数据流图**：从用户操作到网络再回到 UI。  
- [ ] 能说出 MVVM 与 MVI 在你业务里的 **取舍**，而非背定义。  
- [ ] PR 能被自己 **按清单评审**（命名、边界、测试）。

## L3 资深（3–6 年）：模块化、性能、工程化（对照 §2.4 / §2.5 / §3）

### L3 学习目标

- 主导或深度参与 **模块化 / 组件化**（多 module / SPM packages / XCFramework）。  
- 建立 **性能与稳定性** 指标：启动、卡顿、Crash、ANR/Watchdog、包体积。  
- 搭建或改进 **CI/CD**：PR 检查、单元测试、归档、内测分发。  
- 对 **另一端或 Flutter/RN** 达到「能评审方案、知坑」级别。

### L3 — 模块化（对照 §6 考点 1）

**学习路径**

1. 识别 **稳定依赖方向**：基础设施 → 业务基础 → 业务模块 → App Shell。  
2. 抽 **接口模块**（API module）：对外仅暴露接口与模型，实现可替换。  
3. 治理 **循环依赖**：依赖图工具（Gradle dependency analysis / Xcode build graph）。  

**实践里程碑**

- [ ] 把「登录 / 账户」或「支付」抽成独立模块，App 仅组装。  
- [ ] 输出 **模块边界文档**：谁依赖谁、禁止依赖列表。  
- [ ] 大型 MR 采用 **渐进迁移**（Strangler Fig），每步可运行。

### L3 — 性能与稳定性（对照 §2.4）

**Android**

| 方向 | 做什么 | 工具/产出 |
|------|--------|-----------|
| 启动 | 异步初始化、懒加载、避免主线程 IO | SysTrace / App Startup / 自定义打点 |
| 列表 | RecyclerView/Compose Lazy 复用、diff | Choreographer 丢帧分析 |
| ANR | 主线程锁、IPC | traces.txt / Perfetto |
| 包体积 | R8、资源混淆、动态特性模块评估 | APK Analyzer |

**iOS**

| 方向 | 做什么 | 工具/产出 |
|------|--------|-----------|
| 启动 | dyld、静态初始化、`+load` 审计 | Instruments Time Profiler |
| 列表 | Cell 复用、图片解码尺寸 | Core Animation FPS |
| Watchdog | 主线程卡死 | 崩溃日志 + MetricKit |
| 包体积 | Swift 优化级别、Asset Catalog、剥离无用资源 | App Thinning 报告 |

**共性**

- 建立 **性能预算**：例如冷启动 P90 &lt; X ms（与产品对齐）。  
- 上线前 **对比基准**（Baseline vs MR）。

### L3 — 工程化与交付（对照 §2.5）

- CI：`lint` + `unit tests` + （可选）UI 冒烟；失败阻断合并。  
- 多渠道：`dev/staging/prod` 配置隔离（环境变量 / xcconfig / build flavor）。  
- **Feature Flag**：远程开关 + 本地默认值 + 版本兜底逻辑。  
- 灰度：与后端/运维对齐 **放量策略**；客户端 **上报版本与开关命中**。

### L3 — 跨端 / 另一端广度（对照 §2.2）

择一深入理解即可：

- **Flutter**：Widget 树与渲染管线、`Platform Channel`、isolate、shader/jank。  
- **React Native**：Bridge / JSI、Hermes、原生模块边界。  
- **KMP**：expect/actual、与原生 UI 集成策略。

**里程碑**：能写 **1 页选型对比**（团队规模、招聘、性能、风险）。

### L3 — 可观测性入门（对照 §3）

- **埋点**：事件命名规范、公共属性、采样率、隐私字段脱敏。  
- **日志**：分级、关联 request-id、禁止敏感信息明文。  
- **Crash**：符号表、dSYM / mapping；聚类与指派流程。

### L3 产出物

- **性能专项报告** 1 份（含前后对比与后续 backlog）。  
- **CI 流水线说明**（触发条件、缓存策略、产物）。  
- **事故复盘** 参与模板（即使中小事故也可练习）。

### L3 自测标准（对齐 §6 面试）

- [ ] 能讲清一次 **启动优化** 的度量方法与副作用。  
- [ ] 能设计 **网络层**：超时、重试、幂等、缓存一致性的边界。  
- [ ] 能主持 **30 分钟模块化评审**，指出依赖违规。

## L4 架构（6–10 年+）：选型、演进与组织协同（对照 §3 / §4）

### L4 学习目标

- 主导 **技术选型** 与 **迁移路线图**（含回滚与并行运行期）。  
- 输出 **RFC / ADR**；推动 **规范与组件库** 与 Design System 对齐。  
- 处理 **复杂域**：离线、同步、音视频、支付等 **容错与一致性** 边界。  
- **带人**：中级工程师培养计划、PR 标准与架构守门机制。

### L4 — 技术演进（对照 §3）

**方法论**

1. **问题陈述**：业务指标 + 技术痛点 + 若不做的成本。  
2. **方案备选**：至少 2 套，含权衡矩阵（性能、成本、风险、团队）。  
3. **迁移计划**：里程碑、feature flag、观测指标、回滚条件。  
4. **收尾**：文档沉淀、培训、删除旧路径代码（忌永久双轨）。

**复杂域检查清单（示例）**

| 域 | 架构师需澄清的问题 |
|----|-------------------|
| 离线 | 冲突解决策略、最终一致性、队列上限与丢弃策略 |
| 同步 | 增量协议、时钟与版本向量、后台唤醒限制 |
| 音视频 | 弱网、编码档位、前后台切换、功耗 |
| 支付 | 幂等、回调验签、超时与用户提示文案 |

### L4 — 规范与 API 契约（对照 §3）

- 推动 **OpenAPI / GraphQL Schema** 的版本策略与 Review 流程。  
- **破坏性变更**：提前通知、并行字段期、客户端兜底时间表。  
- 组件库：**无障碍**、主题 Token、废弃 API 生命周期。

### L4 — 可观测性体系（对照 §3）

- **SLI/SLO** 意识：崩溃率、启动成功率、核心接口成功率。  
- **仪表盘**：版本维度、地区维度、实验维度。  
- **复盘文化**：无惩罚根因分析 + 行动项 owner + 截止日期。

### L4 — 软技能（对照 §4）

- **向上**：用业务语言解释技术债利息与投资回报。  
- **横向**：后端超时、重试风暴、缓存击穿等 **联合演练**。  
- **向下**：辅导「方案拆解」与「文档先行」习惯。

### L4 产出物

- **年度技术路线图**（与业务 OKR 对齐）。  
- **至少 3 份 ADR**（选题可含：网络栈升级、模块化边界、跨端选型）。  
- **组件库 / 脚手架** 的 adoption 指标（使用率、Issue 闭环时间）。

### L4 自测标准

- [ ] 你能 **否决** 一个不合适的「追新技术」提案，并给出 **数据化理由**。  
- [ ] 重大故障能在 **24h 内** 产出对外可用的复盘摘要（合规前提下）。  
- [ ] 团队 **新人上手文档** 由你维护迭代过一版。

## 专题补强矩阵（按岗位要求选修）

### A. 安全与合规强化（对照 §2.6）

| 项 | 行动 |
|----|------|
| 权限最小化 | 梳理 Manifest / Info.plist，删除无用权限 |
| 传输安全 | Certificate Pinning POC（理解轮换与故障） |
| 存储 | Keychain/Keystore 分层；Biometrics 绑定场景 |
| 逆向认知 | 静态扫描工具报告解读（不必亲自破解） |
| 合规 | 参与隐私影响评估（PIA）流程 |

### B. 双原生 + 大团队（对照附录 7）

- 统一 **分支策略** 与 **合并门禁**。  
- **二进制产物**（Android AAR / iOS XCFramework）发布规范。  
- **Design Token** 两端对齐与自动化校验。

### C. Flutter / RN 为主团队（对照附录 7）

- 引擎升级 playbook（breaking changes 清单）。  
- **原生插件** 质量门禁（线程模型、内存、异常穿透）。  
- 包体积与启动 **专项预算**。
